# ccc-framework（Cocos Creator）初始化设计稿

日期：2026-04-15  
目标：初始化一个可在 Cocos Creator 中“导出资源包”交付的脚本框架，作为后续长期维护的目录与规则基线。

## 背景与约束

- **引擎/编辑器**：Cocos Creator 3.8.8（项目内 `tsconfig.json` 继承 Creator 生成的 `temp/tsconfig.cocos.json`）。
- **交付方式**：对外以 **Creator 资源包**交付，接收方在 Creator 中导入后即可使用。
- **交付内容范围**：资源包 **只包含 TypeScript 脚本**，不包含 prefab/贴图/音频等美术资源。
- **导出根目录**：统一以 `assets/framework/` 作为资源包根目录。
- **导入后稳定性**：尽可能降低与对方工程冲突风险，保持命名空间清晰与依赖方向可控。

## 目标（Goals）

- 建立清晰的框架分层与依赖方向，保证可演进与可维护。
- 统一 import 规则与命名规范，减少跨模块深相对路径导致的重构成本。
- 明确对外交付边界与导出流程，保证对方导入后可稳定编译运行。

## 非目标（Non-goals）

- 本阶段不引入编辑器侧扩展（`editor` 脚本、菜单、自动化面板等）。
- 本阶段不定义具体业务玩法模块（`gameplay` 仅预留，可为空）。
- 本阶段不引入 prefab/UI 资源（`ui/` 仅为代码层框架）。

## 目录结构（最终基线）

框架资源包根目录为：

- `assets/framework/`
  - `base/`（框架基础设施层：框架语义强，规则最严格）
    - `app/`：框架入口/启动配置（初始化编排、全局开关、版本信息）
    - `lifecycle/`：生命周期钩子与调度（启动/暂停/恢复/退出等）
    - `event/`：事件总线/消息机制（优先 typed events）
    - `context/`：全局上下文（运行时状态、服务容器句柄、环境信息聚合）
    - `time/`：计时器/节流去抖/帧循环抽象
    - `di/`：依赖注入/服务注册（轻量 ServiceLocator/Container）
    - `env/`：平台与环境判断（平台适配、debug/release、渠道、特性开关读取）
    - `decorators/`：**带框架语义**装饰器（注册/注入/生命周期声明等）
  - `utils/`：纯 TS 通用工具（包括“纯装饰器工具”）
  - `storage/`：本地存储抽象与实现
  - `net/`：网络抽象与实现
  - `res/`：资源加载门面（对接 `assetManager`）
  - `ui/`：仅代码层 UI 框架（不含 prefab/贴图）
  - `gameplay/`：可选玩法模块（默认不被其它层依赖）

## 模块职责与边界

- `base/*`
  - 定义框架“语义层”的最小集合：启动/生命周期/事件/上下文/DI/环境与平台能力。
  - 只依赖 `utils/*`（不允许依赖其它模块）。
- `utils/*`
  - 与框架语义无关的纯 TS 工具、类型工具、纯装饰器工具等。
  - 不依赖任何其它模块。
- `storage/*`
  - 本地持久化能力：键管理、版本迁移策略接口、不同存储后端实现（可迭代）。
- `net/*`
  - 网络能力：HTTP/WS 薄封装、超时/重试/拦截器、统一错误模型（可迭代）。
- `res/*`
  - 资源加载门面：加载/缓存/释放策略接口与最小实现（可迭代）。
- `ui/*`
  - UI 代码框架：UIManager、界面栈、打开/关闭与事件组织方式（不携带资源）。
- `gameplay/*`
  - 玩法/业务示例模块（可选），只允许“向下”依赖其它层，禁止其它层依赖它。

## 依赖方向规则（强约束）

### 允许的依赖

- `utils`：不依赖任何模块
- `base` → `utils`
- `storage/net/res/ui` → `base`、`utils`
- `gameplay` → 允许依赖 `storage/net/res/ui/base/utils`

### 禁止的依赖

- `utils` 依赖任何模块（禁止）
- `base` 依赖 `storage/net/res/ui/gameplay`（禁止）
- `storage/net/res/ui` 依赖 `gameplay`（禁止）
- 任何模块形成环依赖（禁止）

## import 规则（@fw 别名）

### 约定

- 统一别名：`@fw/*` → `assets/framework/*`
- 框架内部跨模块引用 **必须** 使用 `@fw/...`（禁止使用跨层深相对路径）。

### 对接方式（TS 配置）

- 在项目 `tsconfig.json`（或其可覆盖层）中配置 `compilerOptions.paths`，确保 `@fw/*` 可解析。
- 由于 Creator 可能生成/覆盖部分 tsconfig，本项目将以“在 `tsconfig.json` 的自定义区域补充 paths”为准。
- 对方项目导入资源包后，需同样保留该 paths 约定；若对方不愿引入别名，可提供“降级方案”将 `@fw` 批量替换为相对路径（不作为默认）。

## 命名与导出边界

- 对外导出/交付的唯一根目录：**`assets/framework/`**
- 目录命名：小写、短横线或小驼峰二选一（本仓库以小写 + 目录名固定为主）。
- 文件命名：建议 `PascalCase.ts`（类/组件）与 `camelCase.ts`（函数集合）二者按团队习惯选定；在实现计划阶段最终拍板。
- `base/*` 中对外暴露的 API 要更稳定，破坏性改动需记录变更说明。

## Creator 资源包导出/交付流程（规范）

> 目标：把框架交付为“可导入的 Creator 资源包”，避免混入非必要目录。

- **导出范围**：仅导出 `assets/framework/`。
- **导出前检查（必须）**
  - 确保 `assets/framework/` 下不存在 prefab/贴图/音频等非脚本资源（本阶段约束）。
  - 确保所有脚本可通过 TypeScript 编译（别名 `@fw` 可解析）。
  - 确保没有环依赖与跨层违规依赖（后续可用脚本校验，暂以 code review 为准）。
- **交付物**：导出的资源包文件（或对方认可的压缩包形式）。
- **接入方要求**
  - 导入后确保其项目配置中存在 `@fw/*` 的路径别名解析。

## 质量与一致性（初始规则）

- **代码风格**：优先使用 TypeScript 严格类型（后续实现阶段再决定是否开启 `strict: true`；目前项目 tsconfig 显示 `strict: false`）。
- **边界优先**：新增代码必须先判断属于哪个模块；不清楚时优先放在更“上层”（如 `storage/net/res/ui`），避免污染 `base/utils`。
- **无隐式副作用**：`base/app` 的启动入口应显式触发初始化，避免 import 即执行的副作用。

## 风险与缓解

- **风险：接入方 tsconfig/路径别名不一致导致编译失败**
  - **缓解**：把 `@fw` 约定写入交付说明；如对方无法配合，提供一次性替换为相对路径的降级方案（脚本化，后置）。
- **风险：模块边界被破坏导致演进困难**
  - **缓解**：强依赖方向规则；后续引入脚本/CI 校验（在实现计划阶段定义）。
